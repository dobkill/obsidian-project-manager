# 02. 数据存储与模型

## 1. 存储目录

插件数据默认存放在：

```text
project-manager-data/
```

目录结构：

```text
project-manager-data/
├── config.json
├── projects.json
├── progress-pages.json
└── tasks/
    ├── 2026-05.json
    ├── 2026-06.json
    └── ...
```

数据目录路径可在设置页修改，规则如下：

1. 目标目录已有插件数据且现有插件文件格式正确时，直接加载目标目录中的数据。
2. 目标目录不存在、为空，或插件数据文件格式错误时，会用当前已加载数据在目标目录创建一套新的插件数据文件。
3. 目标目录中存在非插件文件时，为避免误覆盖用户文件，会拒绝切换。
4. `config.json` 中的 `dataFolder` 不会把加载路径反向改回旧目录；实际使用路径以设置页选择的目录为准。

## 2. 配置模型

`config.json` 关键字段：

```ts
type PluginConfig = {
  version: string;
  dataFolder: string;
  overviewTab1Name: string;
  overviewTab2Name: string;
  weekStartsOn: "monday";
  timeSlotMinutes: number;
  heatmapRange: "12months";
  showCompletedTasks: boolean;
  defaultTaskDurationMinutes: number;
  defaultTaskStartTime: string;
};
```

其中：

1. `defaultTaskStartTime` 用于当天没有任何排期任务时的默认开始时间。
2. `defaultTaskDurationMinutes` 用于自动推算结束时间。

## 4. 任务模型

当前任务模型已经升级，不兼容旧版多实例重复任务结构。

```ts
type Task = {
  id: string;
  kind: "simple" | "composite";
  title: string;
  description?: string;
  projectId?: string;
  date: string;
  startTime?: string;
  endTime?: string;
  recurrence: "once" | "daily" | "weekly";
  recurrenceCount?: number | null;
  recurrenceUntil?: string | null;
  subtasks: Array<{
    id: string;
    title: string;
  }>;
  occurrenceDates: string[];
  occurrenceStates: Array<{
    date: string;
    completedAt?: string | null;
    completedSubtaskIds?: string[];
  }>;
  createdAt: string;
  updatedAt: string;
};
```

### 4.1 字段说明

1. `kind` 区分普通任务与组合任务。
2. `date` 表示这条任务系列的起始日期。
3. `occurrenceDates` 记录全部发生日期。
4. `subtasks` 在组合任务中保存子任务定义。
5. `occurrenceStates` 记录每次发生的完成状态；组合任务会把已完成子任务 ID 写入 `completedSubtaskIds`。
6. `recurrenceCount` 和 `recurrenceUntil` 仍然用于描述重复边界。

### 4.2 设计含义

1. 单次任务也是一条系列任务，只是 `occurrenceDates` 只有一天。
2. 普通任务与组合任务共用同一套系列模型，只在完成状态粒度上不同。
3. 重复任务在存储层不再拆成多条实例。
4. 页面展示需要的实例数据，在运行时根据 `occurrenceDates` 展开。

## 5. 运行时实例模型

页面层会把 Task 展开为临时 TaskOccurrence：

```ts
type TaskOccurrence = {
  id: string;
  taskId: string;
  occurrenceDate: string;
  occurrenceNumber: number;
  kind: "simple" | "composite";
  date: string;
  subtasks: Array<{ id: string; title: string }>;
  completedSubtaskIds: string[];
  progress: number;
  totalSteps: number;
  completedSteps: number;
  completed: boolean;
  completedAt?: string | null;
  // 其余标题、时间、项目字段继承自 Task
};
```

用途：

1. 今日任务页按 `date === 今天` 读取。
2. 周任务图按周内日期读取。
3. 组合任务在今日页与周任务图中按子任务粒度操作。
4. 热度图与趋势图按 `completedAt` 的日期统计，仅在整条实例完成时计入完成数。

## 7. 月度任务文件

每个月一个文件：

```ts

  month: string;

};
```

`month` 由任务起始日期所在月份决定。

## 8. 写入时机

所有会改变数据的操作都必须先完成文件写入，再触发视图刷新：

1. 任务新增、编辑、删除、完成状态和子任务完成状态，会立即写入受影响的 `tasks/YYYY-MM.json`。
2. 项目新增、编辑、删除和进度页排序，会立即写入 `projects.json` 与/或 `progress-pages.json`。
3. 设置变更会立即写入插件数据目录中的 `config.json`，并同步写入 Obsidian 插件设置。
4. 插件卸载时会等待仍在队列中的写入完成，并在非只读状态下写入一次当前数据快照，降低快速关闭时丢失最后一次修改的风险。
5. 写入队列按顺序执行；单次写入失败会向当前操作抛错，但不会让后续保存永久停滞。

## 9. 启动与缓存刷新

当前实现采用“文件存储 + 内存缓存”双层机制：

1. 插件初始化时会读取当前数据目录，把 `config.json`、`projects.json`、`progress-pages.json` 和 `tasks/*.json` 加载到内存。
2. Obsidian workspace 进入 ready 后，会再执行一次从当前数据目录回读，补强启动阶段的缓存同步。
3. 点击“项目总览”或“今日任务”入口时，会先展示当前缓存，再异步回读磁盘并刷新视图。
4. 所有写操作完成后，都会再次从文件回读到内存，确保页面展示以落盘后的结果为准。
5. 目录与文件读取优先使用 Vault 缓存，缓存暂未命中时会回退到 `adapter.stat/read/list` 直接检查磁盘。
6. 数据目录创建采用幂等策略；若目录已存在，不会因为重复创建而报 `folder already exists`。

## 10. 当前选择

当前实现没有兼容旧格式迁移逻辑，原因如下：

1. 重复任务存储结构已经彻底变化。
2. 组合任务需要子任务级完成状态，旧结构表达力不足。
3. 项目进度页的核心诉求就是“重复任务单行展示”。
4. 当前项目仍处于可大改阶段，直接切新结构更清晰。
